SecretGuard Documentation

CLI reference for SecretGuard v1.0.0 — 44 patterns, live API verification, bypass detection.

Installation

npm install -g secretguard

Requires Node.js 16+. After install, secretguard is available globally.

Verify install

secretguard --version
# 1.0.0

Quick Start

# Set up in your repo (installs pre-commit hook + patches .gitignore)
cd your-project
secretguard init

# Scan now
secretguard scan .

# Watch mode — rescan on every file save
secretguard scan --watch .

# Scan only what you're about to commit
secretguard scan --staged

One command setup. secretguard init installs the pre-commit hook, creates .secretguardignore, and adds .env, *.pem, *.key to .gitignore automatically.

Philosophy

SecretGuard is prevention-first. Most tools detect secrets after the fact. SecretGuard blocks the commit:

  1. Pre-commit hook — blocks commit if secrets found. Developer must fix before pushing.
  2. Post-commit bypass detection — if developer uses --no-verify, post-commit hook scans the committed diff and fires team notifications immediately.
  3. CI/CD scan — catches anything that slipped through in PRs and merges.
  4. History audit — secretguard audit scans all commits for secrets that entered git history at any point.

Raw secret values are never printed. All matches are masked: ghp_****YZ01.

secretguard scan

Scan a directory or file for secrets.

secretguard scan [path] [flags]
pathDirectory or file to scan. Default: .
--stagedScan only git staged files (used by pre-commit hook)
--last-commitScan the last committed diff — catches --no-verify bypasses
--watchWatch mode — rescan on file save, debounced 300ms
--fail-on=critical,highExit 1 if severity matches. Default: critical
--format=table|json|sarifOutput format. Default: table
--output=file.sarifWrite output to file instead of stdout
--fixShow remediation steps after scan
--no-verifySkip live API verification (faster, offline)
--quietSuppress banner and status messages (for scripts)

Examples

secretguard scan .
secretguard scan src/config.js --fix
secretguard scan --staged --fail-on=critical,high
secretguard scan . --format=json | jq '.findings[].severity'
secretguard scan . --format=sarif --output=results.sarif
secretguard scan --watch src/

secretguard audit

Scan the full git commit history. Finds secrets that were committed and later deleted — they remain accessible in git history.

secretguard audit [flags]
--max-commits=500Limit commits scanned. Default: 500
--since="6 months ago"Only scan commits since date
--format=table|json|sarifOutput format
--fixShow remediation steps

Secrets in git history remain accessible even after deletion from working tree. Use git-filter-repo or BFG Repo Cleaner to purge them, then rotate all affected keys.

secretguard hook

Manage the pre-commit and post-commit git hooks. secretguard init calls this automatically.

secretguard hook install      # installs pre-commit + post-commit hooks
secretguard hook uninstall    # removes hooks (only if installed by SecretGuard)

Two hooks installed. pre-commit blocks commits with secrets. post-commit scans the committed diff and alerts the team if --no-verify was used to bypass.

secretguard init

One-command setup for a new repo.

secretguard init

Does three things:

  1. Installs pre-commit and post-commit hooks via secretguard hook install
  2. Creates .secretguardignore with sensible defaults
  3. Appends missing entries (.env, *.pem, *.key, .aws/credentials) to .gitignore

secretguard fix

Show actionable remediation steps for all detected secrets.

secretguard fix [path]

Each finding gets numbered steps specific to its provider — not generic advice. Example for a GitHub token:

  GitHub Personal Access Token
    1. Go to GitHub → Settings → Developer settings → Personal access tokens
    2. Delete or regenerate this token immediately
    3. Use: export GITHUB_TOKEN=<new-token> or store in .env (gitignored)
    4. Consider using GitHub Actions secrets for CI workflows

secretguard baseline

Suppress existing findings that are false positives or already queued for rotation. New findings after the baseline is created are still blocked.

secretguard baseline generate [path]   # scan and create .secretguardbaseline
secretguard baseline status             # list suppressed findings
secretguard baseline clear              # remove baseline (re-report everything)

The .secretguardbaseline file stores fingerprints (file + line + rule + masked match), not raw secret values. Commit it to your repo so the whole team shares the same suppressions.

Baseline ≠ "it's fine". Baselined secrets still need to be rotated. The baseline gives you time to do it without blocking every developer on every commit.

secretguard ci

Print a ready-to-use CI/CD config template.

secretguard ci github-actions
secretguard ci gitlab-ci
secretguard ci pre-commit-config

Live API Verification

SecretGuard makes safe read-only API calls to confirm if detected credentials are still active:

  • GitHub — GET /user
  • Slack — auth.test
  • Stripe — GET /v1/charges with 0 limit
  • Google — Maps Geocoding API probe

Use --no-verify to skip (faster, works offline). Results show inline:

  VERIFIED ACTIVE — revoke immediately
  ✓ Verified inactive (key revoked or invalid)

Bypass Detection

When a developer uses git commit --no-verify, the pre-commit hook is skipped. SecretGuard installs a post-commit hook that runs after every commit and scans the committed diff.

If secrets are found in a --no-verify commit, the post-commit hook:

  1. Prints a loud warning with BYPASS DETECTED banner
  2. Fires Slack/webhook/email notification with "event": "bypass_commit"
secretguard scan --last-commit    # manually trigger bypass check

Blast Radius

Every finding shows what an attacker can do with that credential — not just "this looks like a key":

  impact:  Full access to Stripe account: create charges on customer cards,
           exfiltrate all payment data, issue refunds, create payouts to
           attacker bank account, access PII of all customers.

Watch Mode

Real-time scanning as you edit files.

secretguard scan --watch .
secretguard scan --watch src/

Rescans on every file save, debounced 300ms. Shows timestamp per change:

  ✓ [14:32:01] config.js — clean
  ⚠ [14:32:45] auth.js — 1 finding (GitHub token)

Notifications

Set environment variables to alert your team in real time:

# Slack
export SECRETGUARD_SLACK_WEBHOOK=https://hooks.slack.com/services/...

# Custom webhook (PagerDuty, internal tooling, etc.)
export SECRETGUARD_WEBHOOK_URL=https://your-server.com/alerts
export SECRETGUARD_WEBHOOK_TOKEN=optional-bearer-token

# Email (direct SMTP — no relay service needed)
export SECRETGUARD_SMTP_HOST=smtp.gmail.com
export SECRETGUARD_SMTP_PORT=587
export SECRETGUARD_SMTP_USER=you@gmail.com
export SECRETGUARD_SMTP_PASS=your-app-password
export SECRETGUARD_EMAIL_TO=security@yourcompany.com
export SECRETGUARD_EMAIL_FROM=secretguard@yourcompany.com

VS Code Extension

Real-time secret detection in the editor. Scans as you type (debounced 800ms on change, immediate on save).

  • Red/yellow squiggles on secret lines in the editor
  • Problems panel integration with clickable findings
  • Status bar: 🔒 SecretGuard: 3 secrets → red when secrets found
  • Right-click → "SecretGuard: Ignore this finding" → adds to .secretguardignore
  • Command palette: SecretGuard: Scan Workspace

Uses the same 44-pattern engine as the CLI — inline, no subprocess, fast.

GitHub Actions

- name: Install SecretGuard
  run: npm install -g secretguard

- name: Scan for secrets
  run: secretguard scan . --fail-on=critical,high --format=sarif --output=secretguard.sarif

- name: Upload to GitHub Security
  uses: github/codeql-action/upload-sarif@v3
  if: always()
  with:
    sarif_file: secretguard.sarif

GitLab CI

secret-scan:
  image: node:20-alpine
  script:
    - npm install -g secretguard
    - secretguard scan . --fail-on=critical,high
  only:
    - merge_requests
    - main

SARIF / GitHub Code Scanning

SARIF 2.1.0 output integrates with GitHub Code Scanning for inline PR annotations. Each finding gets a fingerprint for deduplication across runs.

secretguard scan . --format=sarif --output=results.sarif

Ignore File

Create .secretguardignore in your repo root to skip files or directories:

# .secretguardignore
test/fixtures/
*.example
*.sample
mock-data/
docs/

The scanner also respects .gitignore automatically. Binary files, node_modules, dist, .next, build, and coverage dirs are always skipped.
